class Campaign < ApplicationRecord
  has_many :donation_options, dependent: :destroy
  has_many :donations, dependent: :destroy

  validates :title, :story, :goal_amount, :slug, presence: true
  validates :slug, uniqueness: true
  validates :goal_amount, numericality: { greater_than: 0 }

  # Pending donations count toward progress: this demo has no payment
  # integration to ever flip them to "paid", so excluding pending would mean
  # the progress bar never visibly moves after a donation is submitted.
  # See README for the full rationale.
  def raised_amount
    donations.where(status: [ :pending, :paid ]).sum(:amount)
  end

  def progress_percentage
    return 0 if goal_amount.zero?

    (raised_amount / goal_amount * 100).round(1)
  end
end

Assistant2026-06-08T09:55:31.089Z

Yes — Rails uses Active Record, an ORM (Object-Relational Mapper) built into the framework. It maps database tables to Ruby classes and rows to instances, so you work with Ruby objects instead of writing SQL directly.

Where things live (Rails convention-over-configuration):

  • Models (the Ruby classes that wrap DB tables): app/models/
  • app/models/campaign.rb → Campaign class → campaigns table
  • app/models/donation_option.rb → DonationOption → donation_options table
  • app/models/donation.rb → Donation → donations table

  • All inherit from ApplicationRecord (app/models/application_record.rb), which itself inherits from ActiveRecord::Base — that's where the ORM machinery comes from.

  • Migrations (versioned scripts that create/alter DB tables): db/migrate/

  • e.g. db/migrate/20260608095153_create_campaigns.rb defines the campaigns table's columns.

  • Running bin/rails db:migrate applies them in order against the actual Postgres database.

  • Schema snapshot (the current DB structure as Ruby/SQL, regenerated after each migration): db/schema.rb

  • Controllers (handle HTTP requests, talk to models, render views): app/controllers/

  • e.g. app/controllers/home_controller.rb

So the flow is: a migration creates the table → Active Record introspects that table's columns at boot time → the model class (e.g. Campaign) automatically gets attributes (title, goal_amount, etc.) and query methods (Campaign.find, campaign.donations, Campaign.where(...)) without you writing any SQL or declaring the columns in the class itself — that's the "magic" of the ORM. The associations (has_many :donations), validations, and enums I added in the model files layer Ruby-level behavior on top of that.
