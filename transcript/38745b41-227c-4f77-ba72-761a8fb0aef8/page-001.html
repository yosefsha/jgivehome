Here's how to get it live on Render using the render.yaml blueprint I just added:

1. Connect GitHub to Render

  1. Go to dashboard.render.com and sign up/log in (GitHub OAuth is the easiest option — it does the account linking for you).
  2. If you used email/password instead: go to Account Settings → GitHub and click Connect account, then authorize Render to access your repos (you can grant access to just jgivehome or all repos).

2. Deploy from the Blueprint

  1. On the dashboard, click New → Blueprint.
  2. Select the yosefsha/jgivehome repo (search for it in the list — Render scans repos it has access to).
  3. Render will detect render.yaml at the repo root and show you a preview: a web service (jgivehome, built from the Dockerfile) and a Postgres database (jgivehome-db).
  4. Click Apply — Render provisions both.

3. Set the one secret it can't infer

The blueprint marks RAILS_MASTER_KEY as sync: false, meaning Render won't store it in the YAML (it's sensitive — keeps it out of git). You add it manually: 1. Open the file config/master.key locally (it's gitignored, so only you have it) and copy its contents. 2. In the Render dashboard, go to the jgivehome web service → Environment tab. 3. Add an env var RAILS_MASTER_KEY with that value, and save.

4. Watch the first deploy

  • Render will build the Docker image (this runs assets:precompile/Tailwind build as part of the Dockerfile's asset pipeline stage) and then boot the container.
  • bin/docker-entrypoint runs ./bin/rails db:prepare on boot, which creates/migrates the database — so the first deploy also sets up the schema against jgivehome-db automatically.
  • DATABASE_URL is wired automatically via the fromDatabase reference in render.yaml, pointing at the managed Postgres connection string.

5. Confirm it's live

Once the deploy shows "Live," open the service's *.onrender.com URL — you should see "Hello, world!" rendered from HomeController#index.

Note: Render's free-tier web services spin down after ~15 minutes of inactivity, so the first load after idling can take ~30-60 seconds to wake up — that's expected, not a bug.

Let me know once it's deployed and I'll help troubleshoot if anything doesn't come up cleanly.

In the Render dashboard:

  2. In the left sidebar of that service, click Environment (not "Environment Groups" — that's for sharing vars across multiple services).

  5. Key: RAILS_MASTER_KEY
  6. Value: the contents of your local config/master.key file
  7. Click Save Changes — Render will prompt you that this triggers a redeploy, which is what you want.

"Environment Groups" is an optional feature for sharing the same set of vars across several services — you don't need it for a single service like this; adding the var directly on the service's Environment tab is the right place.
