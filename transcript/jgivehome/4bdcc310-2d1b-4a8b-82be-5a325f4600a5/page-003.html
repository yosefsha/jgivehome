Here's the step-by-step causal chain of the "missing video" bug:

1. Initial deploy — before video_url existed The orange-garden campaign was first created on Render via db:seed (which the entrypoint runs on every boot) at a point when the campaigns table had no video_url column yet — that column was added later by migration 20260608121605_add_video_url_to_campaigns.rb. So the very first row for slug: "orange-garden" was persisted without a video_url.

2. The seed code uses find_or_create_by!

orange_garden = Campaign.find_or_create_by!(slug: "orange-garden") do |campaign|
  campaign.title = "הגן הכתום"
  ...
  campaign.video_url = "https://www.youtube.com/watch?v=4Z_xXXR3ddU"
end

The crucial Rails behavior: the block passed to find_or_create_by! only executes when a new record is created. If a row matching slug: "orange-garden" already exists, Rails just returns that row — the block (including the video_url = line) never runs.

3. Every subsequent boot just finds the existing row Once the column was added and the campaign already existed, every later deploy's db:seed run found the pre-existing row, skipped the block entirely, and left video_url as NULL in the database — forever, no matter how many times db:seed ran.

4. The view correctly checks video_url.present?

<% if video_url.present? %>
  <%= link_to video_url, ... do %> ... watch the video ... <% end %>
<% end %>

This check (the line you selected) was actually working correctly — video_url really was blank (nil), so .present? correctly returned false and the link was correctly suppressed. The bug wasn't in this conditional; it was upstream, in why the data was missing.

5. The fix — explicit backfill Since find_or_create_by!'s block can't retroactively populate a column added after the row existed, I added an explicit update right after it:

orange_garden.update!(video_url: "https://www.youtube.com/watch?v=4Z_xXXR3ddU") if orange_garden.video_url.blank?

This runs on every db:seed (every boot), is a no-op once video_url is set, and — critically — fixes the existing production row the next time the app deploys and boots, rather than only affecting freshly-created databases.

I verified this locally by forcing a campaign's video_url to nil (simulating production's exact state) and re-running the seed — it correctly backfilled the value (commit 986a0a5, already pushed).
