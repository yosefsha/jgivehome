The spec/ directory holds the project's automated tests, written using RSpec (the testing framework we set up earlier). It mirrors the role test/ plays in default Rails apps, but with RSpec's describe/it/expect syntax instead of Minitest's.

How it fits into the dev process:

  1. Safety net for changes — every time you run bundle exec rspec, it boots a test environment, runs every *_spec.rb file it finds, and tells you what passes/fails. So when you add a feature or refactor later, you immediately know if you broke something that used to work.

  2. Specification as documentation — the describe/it blocks read like sentences ("Donation validates presence of donor_email"), so the spec files double as a readable description of what the code is supposed to do — useful for you (or anyone else) coming back to the code later.

  3. Conventional structure mirrors app/:

  4. spec/models/ — tests for app/models/*.rb (validations, associations, custom methods like raised_amount)
  5. spec/requests/ — tests that simulate real HTTP requests through the router → controller → view (like the home_spec.rb we wrote for GET /)
  6. spec/factories/ — FactoryBot definitions (blueprints for quickly building test records, e.g. create(:campaign), instead of hand-writing Campaign.new(title: "...", story: "...", ...) in every test)

  7. spec/rails_helper.rb / spec/spec_helper.rb — shared setup/configuration loaded before specs run

  8. CI gate — typically wired so that a PR/deploy can't go out unless the spec suite is green, catching regressions before they reach production (this project doesn't have CI configured yet, but it's the natural next step).

In short: app/ is "what the code does," spec/ is "proof that it does what it's supposed to, that you can re-run any time."

To rerun the test suite, use:

bundle exec rspec

A few useful variations:

  • Run a single file: bundle exec rspec spec/models/campaign_spec.rb
  • Run a single example by line number: bundle exec rspec spec/models/campaign_spec.rb:12
  • Run only the previously failed examples: bundle exec rspec --only-failures
  • Run a folder: bundle exec rspec spec/models/

bundle exec ensures you're using the gem versions locked in this project's Gemfile.lock rather than whatever's installed globally — that's why we don't just run rspec directly.
