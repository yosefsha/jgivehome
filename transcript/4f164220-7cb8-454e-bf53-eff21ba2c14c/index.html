Running the tests

bundle exec rspec

That runs everything under spec/ — model, request, and system specs together.

What's used, and why

  • RSpec (rspec-rails) — the test framework itself: describe/it/expect syntax, organizes specs into spec/models, spec/requests, spec/system.
  • FactoryBot (factory_bot_rails) — builds test Campaign/DonationOption/Donation records (spec/factories/) without hand-writing setup boilerplate or relying on fragile fixtures.
  • Capybara + Selenium WebDriver (headless Chrome) — drives the browser for system specs: spec/support/capybara.rb configures driven_by(:selenium, using: :headless_chrome, screen_size: [1400, 1400]). These are needed because key behaviors here are JS-driven — Stimulus-powered tabs/modal, and Turbo Stream broadcasts that update the page live without a reload — and a request spec can't observe any of that; only a real (headless) browser can.

Spec breakdown

  • spec/models — validations, enum behavior, and Campaign#raised_amount/#progress_percentage (including that pending counts toward progress).
  • spec/requests — e.g. the campaign show page renders title, raised amount, goal, and progress bar correctly for seeded data.
  • spec/system — full browser-driven flows: switching tabs, opening the donation modal and submitting it (with validation-error display), and the live-update spec that runs two concurrent Capybara sessions to prove a donation submitted in one session broadcasts the updated raised amount and new donation entry to the other live, with no reload.

The choice of headless Chrome over a plain rack-test driver was necessary specifically because Turbo Streams and Stimulus controllers only execute in a real JS engine — that's the layer where this app's most distinctive behavior (live progress updates) actually lives, so it had to be tested at that level rather than mocked out.
